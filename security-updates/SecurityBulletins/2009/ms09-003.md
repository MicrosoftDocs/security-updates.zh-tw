Microsoft Security Bulletin MS09-003 - 重大
===========================================

Microsoft Exchange 中的弱點可能會允許遠端執行程式碼 (959239)
------------------------------------------------------------

發行: 2009年2月11日 | 更新: 2009年3月2日

**版本:** 2.1

### 一般資訊

#### 提要

這個安全性更新可解決 Microsoft Exchange Server 中兩項未公開報告的弱點。 第一弱點是當蓄意製作的 TNEF 訊息被傳送至 Microsoft Exchange Server 時，可能會允許遠端執行程式碼。 成功利用此弱點的攻擊者可以 Exchange Server 的 服務帳戶權限，取得受影響系統的完整控制權。 第二弱點是當蓄意製作的 MAPI 命令被傳送至 Microsoft Exchange Server 時，可能會導致拒絕服務。 成功利用此弱點的攻擊者可造成使用 EMSMDB32 提供者的 Microsoft Exchange System Attendant 服務及其他服務停止回應。

對於所有受支援版本的 Microsoft Exchange 2000 Server、Microsoft Exchange Server 2003、Microsoft Exchange Server 2007 及 Microsoft Exchange Server MAPI Client，此安全性更新的等級為「重大」。 如需更多資訊，請參閱本節中的＜受影響及不受影響的軟體＞小節。

此安全性更新會修改 Microsoft Exchange Server 解譯 TNEF 訊息和 MAPI 命令的方式，以解決這項弱點。 如需更多有關弱點的資訊，請參閱下節＜弱點資訊＞下的特定弱點項目的＜常見問題集 (FAQ)＞小節。

**建議。** Microsoft 建議客戶立即套用此更新。

**已知問題。** [Microsoft 知識庫文件編號 959239](https://support.microsoft.com/kb/959239) 詳細記錄了目前安裝此安全性更新可能發生的已知問題，並提供這些問題的建議解決方案。 目前已知問題和建議解決方案僅適用於此軟體的特定版本，本文會提供詳細文章的連結。

#### 受影響及不受影響的軟體

下列軟體已經過測試判斷哪些版號或版本會受到影響。 其他版本超過它們的支援週期或不受影響。 若要瞭解您的軟體版本的支援週期，請造訪 [Microsoft 產品技術支援週期網站](https://go.microsoft.com/fwlink/?linkid=21742)。

與本安全性更新相關的常見問題集 (FAQ)  
------------------------------------
  
 
**為何於 2009 年 2 月 16 日重新發行此安全性公告？**    
Microsoft 重新發行此公告是為了將 Microsoft Exchange Server MAPI Client 新增為受影響的軟體。 其他更新套件不受此重新發行版本影響。 執行其他所有受支援或受影響版本的 Microsoft Exchange Server 的客戶，若已成功套用原來的安全性更新套件，則不必再以原始更新套件進行任何進一步的動作。
  
Microsoft 另外還新增一些項目至＜與本安全性更新相關的常見問題集 (FAQ)＞章節，內容與更新 MAPI Client 和 Exchange 系統管理工具相關。
  
**我所使用的是 Microsoft Exchange Server MAPI Client 的舊版本，我該如何更新至 6.5.8069 版？**    
Microsoft 已為 Microsoft Exchange Server MAPI Client 使用者提供免費更新至 6.5.8069 版，可從[下載中心](https://www.microsoft.com/download/details.aspx?familyid=e17e7f31-079a-43a9-bff2-0a110307611e)取得。 因此，所有使用舊版 Microsoft Exchange Server MAPI Client 和 Collaboration Data Objects 1.2.1 套件的客戶應該更新至 6.5.8069 版，以免受此公告中描述的弱點影響。
  
**適用於 Microsoft Exchange Server MAPI Client 的更新套件，也包括 CDO 1.2.1。CDO 1.2.1 是否受到本公告中所描述的弱點影響？**    
Microsoft Exchange Server MAPI Client 的下載，也包括 Collaboration Data Objects (CDO)1.2.1 套件。CDO 1.2.1 套件不受到本公告中所列出的弱點影響。 套件已包含在下載中，方便您使用。
  
**我如何知道系統上是否已安裝 Microsoft Exchange Server MAPI Client 和 Collaboration Data Objects 1.2.1？**    
在 \[控制台\] 中，檢查 \[新增或移除程式\] 工具 (Windows Server 2003 及之前的版本)，或是 \[程式和功能\] (Windows Vista 和 Windows Server 2008) 的「Messaging API and Collaboration Data Objects 1.2.1」項目。
  
**下載中心有兩個具有相同的標題的 Microsoft Exchange Server MAPI Client 與 Collaboration Data Objects 1.2.1，可供下載。 這兩者有差別嗎？**    
不，沒有差別。 MAPI Client 可以從下列任一頁面進行下載：  
[Microsoft Exchange Server MAPI Client 與 Collaboration Data Objects 1.2.1](https://www.microsoft.com/download/details.aspx?familyid=e17e7f31-079a-43a9-bff2-0a110307611e&displaylang=en)  
[Microsoft Exchange Server MAPI Client 與 Collaboration Data Objects 1.2.1](https://www.microsoft.com/download/details.aspx?familyid=94274318-27c4-4d8d-9bc5-3e6484286b1f&displaylang=en)
  
**我有 Exchange Server 2003 的 Exchange 系統管理工具。我的系統是否容易受到影響？**    
Exchange Server 2003 的 Exchange 系統管理工具包含弱點程式碼；然而，只有於伺服器也正在執行 Exchange 服務的使用中執行個體時，才能利用此弱點。 因此，僅執行 Exchange Server 2003 的 Microsoft Exchange 系統管理工具，而且已啟用 Exchange 服務的使用中執行個體之客戶，應該安裝 Exchange Server 2003 Service Pack 2 安全性更新 (KB959897)，以避免受此公告中描述的弱點影響。
  
如果 Exchange 服務已停用、在此安全性公告中所討論的易遭利用版本還未曝露出來，而且客戶還未收到此更新；儘管如此，客戶仍可以安裝 Exchange Server 2003 Service Pack 2 安全性更新 (KB959897)，以做為深度防禦措施。
  
**我是協力廠商應用程式開發人員，建議客戶將適用於 Exchange Server 2003 的 Exchange 系統管理工具做為必要條件安裝，以使用我的應用程式。 要如何更新這些程式？**    
僅執行適用於 Exchange Server 2003 的 Exchange 系統管理工具的客戶，應該安裝 Exchange Server 2003 Service Pack 2 安全性更新 (KB959897)，以免受此公告中描述的弱點影響。
  
**我已安裝適用於 Windows Vista 的 Exchange 系統管理員。 這樣的話，我的系統是否容易受到影響？**    
[適用於 Windows Vista 的 Exchange 系統管理員](https://www.microsoft.com/download/details.aspx?familyid=3403d74e-8942-421b-8738-b3664559e46f&displaylang=en)，可下載用於在 Windows Vista 作業系統上，執行適用於 Exchange Server 2003 的 Exchange 系統管理員。 雖然，適用於 Windows Vista 的 Exchange 系統管理員不受此公告中描述的弱點影響，Microsoft Exchange Server MAPI Client 與 Collaboration Data Objects 1.2.1 仍是安裝適用於 Windows Vista 的 Exchange 系統管理員之必要條件。 因此，執行適用於 Windows Vista 的 Exchange 系統管理員的客戶，應該確認其系統上執行的 Microsoft Exchange Server MAPI Client 與 Collaboration Data Objects 1.2.1，已更新至最新的版本。
  
**何處可找到檔案資訊詳細資料？**    
如需檔案資訊詳細資料，請參閱 [Microsoft 知識庫文件編號 959239](https://support.microsoft.com/kb/959239)。
  
**Microsoft Exchange Server 2007 和 Microsoft Exchange Server 2003 的服務模型有什麼不同？而這些差異對於這個安全性公告中的更新有什麼影響？**    
Microsoft Exchange Server 2007 發行時，Microsoft Exchange 已根據客戶意見並考量與其他 Microsoft 產品的一致性，更改為新的服務模型。 Exchange Server 2007 更新在所提供的更新層級和個別檔案層級方面，皆是累計的，但 Exchange Server 2003 更新只有在檔案層級才是累計的。
  
如需 Microsoft Exchange 服務模型的詳細說明，請參閱 [Microsoft Exchange Server 2007 產品說明文件](https://go.microsoft.com/fwlink/?linkid=89577)。 關於新的 Exchange 服務模型如有疑問，請連絡 [Microsoft 產品支援服務](https://support.microsoft.com/common/international.aspx)。
  
**我是否必須按特定順序安裝Exchange Server 2007 伺服器的更新彙總套件？**    
我們的測試基礎結構可幫助確保更新程式會以多種伺服器角色執行。 因此，您不須要以特定順序套用更新彙總套件至執行不同角色的 Exchange 伺服器。 但是，您必須套用更新彙總套件至您環境中的每一個 Exchange Server 2007 伺服器。 這是因為更新彙總套件沒有針對不同 Exchange 角色或特定檔案設定而有不同版本。
  
如果您是「CAS Proxy 部署指南」的客戶，並且您已經部署 CAS-CAS Proxy 處理，請先為網際網路對向的 Client Access Server 套用更新彙總套件，再為非網際網路對向的 Client Access Server 套用更新彙總套件。 對於其他 Exchange Server 2007 組態，套用更新彙總套件至伺服器沒有特定順序。
  
如需更多關於 CAS-CAS Proxy 處理的資訊，請參閱＜瞭解 Proxy 處理與重新導向＞。
  
**為何本次更新中仍然包含數個已回報的安全性弱點？**    
因為解決這些弱點所需的修正作業位於相關檔案中，所以在本次更新中仍然提供對這些弱點的支援。 客戶不需要安裝數個幾乎相同的更新，只需要安裝本更新即可。

本公告所列出的受影響軟體版本已經過測試判斷哪些版本會受到影響。 其他版本超出它們的支援週期。 若要瞭解您的軟體版本的支援週期，請造訪 [Microsoft 產品技術支援週期網站](https://go.microsoft.com/fwlink/?linkid=21742)。

記憶體損毀弱點 - CVE-2009-0098  
------------------------------
  
 
Microsoft Exchange Server 解碼訊息的「傳輸中性封裝格式」(TNEF) 資料的方式，存在遠端執行程式碼的弱點。
  
若要以一般性弱點清單中的標準項目來檢視此弱點，請參閱 [CVE-2009-0098](https://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0098)。
  
#### 記憶體損毀弱點 - CVE-2009-0098 的緩和因素
  
Microsoft 尚未找到此項弱點的任何緩和因素。
  
#### 記憶體損毀弱點 - CVE-2009-0098 的因應措施
  
因應措施指的是無法徹底修正弱點，但有助於在套用更新之前封鎖已知攻擊模式的設定變更。 Microsoft 測試了下列因應措施和狀態，討論因應措施是否會降低功能：
  
-   **封鎖 Microsoft Exchange Server 上的 MS-TNEF 有助於防止透過 SMTP 電子郵件嘗試利用此弱點**
  
    封鎖 application/ms-tnef MIME 內容類型，有助於防止 Exchange 伺服器及其他受影響的程式，在客戶無法安裝可用的安全性更新時嘗試利用此弱點。
  
    系統可設定成封鎖以電子郵件附件形式接收到的某些檔案類型。 Microsoft TNEF 編碼的電子郵件訊息通常稱為 Rich Text 格式 (RTF) 電子郵件訊息，其中可能會包含惡意 OLE 物件。 這類電子郵件訊息含有儲存了 TNEF 資訊的檔案附件。 此檔案附件通常稱為 Winmail.dat。封鎖此檔案，並且封鎖 ms-tnef MIME 類型，有助於防止 Exchange Server 及其他受影響的程式在客戶無法安裝可用的安全性更新時嘗試利用此弱點。 為協助保護 Exchange Server 電腦避免遭到透過 SMTP 發動的攻擊，在 Winmail.dat 檔案及所有 application/ms-tnef MIME 類型內容到達 Exchange Server 電腦之前，就先將其封鎖。
  
    **注意：** 將 Exchange Server 中的 \[Exchange RTF 格式\] 選項設定為 \[絕不使用\]，或是透過編輯登錄機碼來停用 TNEF 處理，仍無法減輕此弱點造成的影響。
  
    **注意：** 對於不受這些因應措施保護的訊息通訊協定 (如 X.400)，Exchange 也給予支援。。 我們建議系統管理員對所有其他用戶端及訊息傳輸通訊協定要求驗證，以協助避免使用這些通訊協定發動攻擊。
  
    **注意：** 僅過濾附件檔名為 Winmail.dat 的郵件，可能無法有效保護您的系統。 惡意檔案附件可能會使用其他可由 Exchange Server 電腦進行處理的檔案名稱。 為協助防範惡意電子郵件訊息，請封鎖所有 application/ms-tnef MIME 類型內容。
  
    封鎖 Winmail.dat 檔案及其他 TNEF 內容的方法有很多種。 以下列出部分建議：
  
    -   您可以使用 ISA Server 2000 SMTP Message Screener 封鎖所有檔案附件或僅封鎖 Winmail.dat 檔案。 如果您使用 ISA Server 2000，封鎖所有檔案附件可為此問題提供最佳保護，因為 ISA Server 2000 不支援根據 MIME 內容類型封鎖內容之功能。 如需更多資訊，請參閱 [Microsoft 知識庫文件編號 315132](https://support.microsoft.com/?id=315132)。  
    -   您可以使用 ISA Server 2000 SMTP Filter 封鎖所有檔案附件或僅封鎖 Winmail.dat 檔案。 如果您使用 ISA Server 2000，封鎖所有檔案附件可為此問題提供最佳保護，因為 ISA Server 2000 不支援根據 MIME 內容類型封鎖內容之功能。 如需更多資訊，請參閱 [Microsoft 知識庫文件編號 320703](https://support.microsoft.com/?id=320703)。  
    -   您可以使用 ISA Server 2004 SMTP Filter 和 Message Screener 封鎖所有檔案附件或僅封鎖 Winmail.dat 檔案。 如果您使用 ISA Server 2004，封鎖所有檔案附件可為此問題提供最佳保護，因為 ISA Server 2004 不支援根據 MIME 內容類型封鎖內容之功能。 如需更多資訊，請參閱 [Microsoft 知識庫文件編號 888709](https://support.microsoft.com/?id=888709)。  
    -   您可以使用 Microsoft Exchange Server 2007 中的附件過濾功能，於伺服器層級進行篩選，以控制使用者收到的附件。 如需詳細資訊，請參閱＜附件過濾＞。  
    -   您可以使用協力廠商電子郵件篩選功能，封鎖所有 application/ms-tnef MIME 類型的內容，以防其傳送至 Exchange Server 電腦或易受影響的應用程式。
  
    **因應措施的影響：** 如果封鎖了 TNEF 附件，將無法正確收到使用 RTF 格式的電子郵件訊息。 在某些情況下，使用者可能會收到空白電子郵件訊息，而不是原始 RTF 格式的電子郵件訊息。 在其他情況下，使用者可能根本不會收到使用 RTF 格式的電子郵件訊息。 封鎖 TNEF 附件不會影響使用 HTML 格式或純文字格式的電子郵件訊息。
  
#### 記憶體損毀弱點 - CVE-2009-0098 的常見問題集
  
**這個弱點的範圍為何？**    
這是遠端執行程式碼的弱點。 成功利用此弱點的攻擊者，可以在使用者開啟或預覽以 TNEF 格式傳送的蓄意製作電子郵件訊息，或是在 Exchange Server Information Store 處理蓄意製作的訊息時，以 Exchange Server 服務帳戶權限，取得受影響系統的完整控制權。 攻擊者接下來能在系統上安裝程式；檢視、變更或刪除資料；或建立新帳戶。
  
**造成這個弱點的原因為何？**    
Exchange 伺服器未能正確解碼 TNEF 格式的訊息。
  
**什麼是 TNEF？**    
傳送中性封裝格式 (TNEF) 是 Microsoft Exchange Server 在傳送 RTF 格式訊息時所使用的格式。 當 Microsoft Exchange 傳送訊息至其他 Microsoft 電子郵件用戶端時，它會解壓縮所有格式設定資訊，並將其編碼成特殊的 TNEF 區塊。 然後將訊息分成兩部分傳送： 已移除格式設定的文字訊息，以及 TNEF 區塊中的格式設定指示。 在接收端，Microsoft 電子郵件用戶端會處理 TNEF 區塊，並將訊息重新格式化。
  
**攻擊者可能會利用這項弱點採取什麼行動？**    
成功利用此弱點的攻擊者可以 Exchange Server 的 服務帳戶權限，取得受影響系統的完整控制權。 攻擊者接下來能在系統上安裝程式；檢視、變更或刪除資料；或建立新帳戶。
  
**攻擊者如何利用這項弱點？**    
攻擊者可傳送蓄意製作的 TNEF 訊息至 Microsoft Exchange Server，以利用此弱點。
  
**因為這個弱點而承受風險的主要系統有哪些？**    
Microsoft Exchanger Server 有受攻擊的風險。
  
**更新的作用何在？**    
更新會修正 Microsoft Exchange Server 解譯特定 TNEF 內容的方式。
  
**當安全性公告發行時，這項弱點是否已被揭發出來？**    
否。 Microsoft 是經由可靠的來源接獲有關這項弱點的訊息。
  
**當本安全性公告發行時，Microsoft 是否已接獲任何消息，指出這項弱點已遭有心人士利用？**    
否。 當本安全性公告初次發行時，Microsoft 並未接到任何有關本弱點已成為公開攻擊媒介的消息，也沒有發現任何以此概念發展的程式碼公開範例。
  
常值處理弱點 - CVE-2009-0099  
----------------------------
  
 
由於 EMSMDB2 (Electronic Messaging System Microsoft Data Base，32 位元版) 提供者處理無效 MAPI 命令的方式，導致拒絕服務弱點。 攻擊者可傳送蓄意製作的 MAPI 命令給使用 EMSMDB32 提供者的應用程式，以利用此弱點。 成功利用此弱點的攻擊者可使應用程式停止回應。
  
由於 Microsoft Exchange System Attendant 也是使用 EMSMDB32 提供者，因此也會受到拒絕服務弱點的影響。 Microsoft Exchange System Attendant 是 Microsoft Exchange 的核心服務之一，並執行各種與 Exchange 系統維護相關的功能。
  
若要以一般性弱點清單中的標準項目來檢視此弱點，請參閱 [CVE-2009-0099](https://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0099)。
  
#### 常值處理弱點 - CVE-2009-0099 的緩和因素：
  
Microsoft 尚未找到此項弱點的任何緩和因素。
  
#### 常值處理弱點 - CVE-2009-0099 的因應措施：
  
Microsoft 尚未找到此項弱點的任何因應措施。
  
#### 常值處理弱點 - CVE-2009-0099 的常見問題集：
  
**這個弱點的範圍為何？**    
這是一項拒絕服務的弱點。 成功利用此弱點的攻擊者可造成使用 EMSMDB32 提供者的 Microsoft Exchange System Attendant 服務或其他服務停止回應。
  
**造成這個弱點的原因為何？**    
Microsoft Exchange Server 處理 EMSMDB32 提供者中的一個命令的方法不正確，因此導致此弱點。
  
**什麼是 MAPI？**  
MAPI 是具備郵件功能和郵件感知功能的應用程式用來建立、操控、傳送及儲存郵件訊息的一組功能。 它為應用程式開發人員提供可定義郵件訊息目的和內容的工具，讓他們能靈活管理儲存的郵件訊息。 MAPI 還為應用程式開發人員提供公用介面，協助他們建立無需仰賴基礎郵件系統，且具備郵件功能和郵件感知功能的應用程式。 如需詳細資訊，請參閱 [MAPI 的 MFC 文件庫參考資料](https://msdn.microsoft.com/en-us/library/245421h5(vs.80).aspx) (英文)。
  
**什麼是 EMSMDB32？**  
Electronic Messaging System Microsoft Data Base，32 位元版 (EMSMDB32) 提供者是指具備 MAPI 傳送與郵件儲存區提供者身分的 Exchange Transport 提供者。 它能讓訊息提送至 Exchange Server，以及將訊息讀入 (甚至可能寫入) Exchange 儲存區中。 如需有關 MAPI 與提供者如何合作的詳細資訊，請參閱 [Outlook、CDO、MAPI 與提供者如何合作](https://technet.microsoft.com/en-us/library/aa996249.aspx)(英文)。
  
**攻擊者可能會利用這項弱點採取什麼行動？**    
成功利用此弱點的攻擊者，可造成使用 EMSMDB32 提供者的應用程式或服務停止回應。 其中一個可能停止回應的服務是 Microsoft Exchange System Attendant 服務，因為它也使用 EMSMDB32 提供者。 Microsoft Exchange System Attendant 執行各種與 Exchange 系統維護和處理相關的功能，例如：產生通訊清單、離線通訊錄和目錄查詢功能。 如果 Exchange System Attendant 服務停止運作，這些工作將會受到影響。
  
**攻擊者如何利用這項弱點？**    
攻擊者可嘗試傳送蓄意製作的 MAPI 命令至 Microsoft Exchange Server，以利用此弱點。 成功利用此弱點的攻擊者可造成郵件服務停止回應。
  
**因為這個弱點而承受風險的主要系統有哪些？**    
使用 EMSMDB 提供者的 Exchange 伺服器系統，受到此弱點攻擊的風險最高。
  
**更新的作用何在？**    
本更新會修改 Microsoft Exchange Server 處理格式錯誤之 IMAP 命令的方式，進而移除此項弱點。
  
**當安全性公告發行時，這項弱點是否已被揭發出來？**    
否。 Microsoft 是經由可靠的來源接獲有關這項弱點的訊息。
  
**當本安全性公告發行時，Microsoft 是否已接獲任何消息，指出這項弱點已遭有心人士利用？**    
否。 當本安全性公告初次發行時，Microsoft 並未接到任何有關本弱點已成為公開攻擊媒介的消息，也沒有發現任何以此概念發展的程式碼公開範例。

    您可以使用 Microsoft Baseline Security Analyzer (MBSA) 工具確認受影響系統上是否已經套用安全性更新程式。 如需詳細資訊，請參閱本公告前段的＜偵測與部署工具及指南＞一節。

-   **檔案版本驗證**

    由於 Microsoft Exchange 版本眾多，您系統上實際執行的步驟可能會與此處描述的不同。 如遇到不同的狀況，請參閱產品的說明文件以完成這些步驟。

    1.  按一下 \[開始\]，然後按一下 \[搜尋\]。
    2.  在 \[搜尋結果\] 窗格中，在 \[搜尋小幫手\] 下按一下 \[所有檔案和資料夾\]。
    3.  在 \[部份或完整的檔案名稱\] 方塊中，輸入適當檔案資訊表中的檔案名稱，再按一下 \[搜尋\]。
    4.  在檔案清單中，用滑鼠右鍵按一下適當檔案資訊表中的某個檔案名稱，再按 \[內容\]。    
        **注意**：視所安裝的作業系統或程式的版本而定，檔案資訊表中列出的檔案未必會全部安裝。
    5.  在 \[版本\] 索引標籤上，比較檔案版本與適當檔案資訊表中記錄的版本，判斷您系統上安裝的檔案版本。    
        **注意：** 在安裝時，檔案版本以外的屬性可能會變更。 若要驗證是否成功套用更新程式，利用檔案資訊表比對其他檔案屬性並不是妥當的做法。 此外，在某些情況下，檔案的名稱在安裝時可能會有所變更。 如果缺少檔案或版本資訊，請採用其他可用的方法來驗證更新程式的安裝情形。

-   **登錄機碼驗證**

    您也可以查看本節「參考表」中所列的登錄機碼，確認此安全性更新所安裝的檔案。
