Microsoft Security Bulletin MS10-041 - 重要
===========================================

Microsoft .NET Framework 中的弱點可能會允許竄改 (981343)
--------------------------------------------------------

發行: 2010年6月8日 | 更新: 2010年6月30日

**版本:** 1.2

### 一般資訊

#### 提要

這個資訊安全性更新可解決 Microsoft .NET Framework 中一項公開揭露的弱點。 此弱點可能會允許已簽署的 XML 內容竄改資料，而且不會被偵測到。 這對自訂應用程式的安全性影響多寡，取決於已簽署的內容是如何用於特定的應用程式中。 如果已簽署的 XML 訊息是透過安全通道 (例如 SSL) 傳輸，便不會受到此弱點影響。

對於適用於 Microsoft Windows 2000、Windows XP、Windows Server 2003、Windows Vista、Windows Server 2008、Windows 7 與 Windows Server 2008 R2 的所有受影響 Microsoft .NET Framework 版本，此資訊安全性更新的等級為「重要」。如需更多資訊，請參閱本節中的＜受影響及不受影響的軟體＞小節。

此資訊安全性更新藉由變更 XMLDsig 建議在 Microsoft .NET Framework 中的實作方式，以解決此弱點。 如需更多有關弱點的資訊，請參閱下節＜弱點資訊＞下的特定弱點項目的＜常見問題集 (FAQ)＞小節。

**建議。**  大部分客戶都已啟用自動更新，並且不必須採取任何行動，因為資訊安全性更新將自動下載和安裝。 沒有啟用自動更新的客戶則必須檢查更新，並手動安裝更新。 如需有關自動更新中特定設定選項的資訊，請參閱 [Microsoft 知識庫文件編號 294871](http://support.microsoft.com/kb/294871/zh-tw)。

若是系統管理員和企業安裝，或是想要手動安裝此資訊安全性更新的使用者，Microsoft 建議客戶使用更新管理軟體，立即套用更新，或使用 [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747) 服務檢查更新。

另請參閱本公告下文的＜偵測與部署工具及指南＞章節。

**已知問題。** [Microsoft 知識庫文件編號 981343](http://support.microsoft.com/kb/981343/zh-tw) 中記載了客戶安裝此資訊安全性更新時，可能會發生的目前已知問題。 該文件並說明了解決這些問題的建議解決方案。

#### 受影響及不受影響的軟體

下列軟體已經過測試判斷哪些版號或版本會受到影響。 其他版本超過它們的支援週期或不受影響。 若要瞭解您的軟體版本的支援週期，請造訪 [Microsoft 產品技術支援週期網站](http://go.microsoft.com/fwlink/?linkid=21742)。

與本資訊安全性更新相關的常見問題集 (FAQ)
----------------------------------------

**為何於 2010 年 6 月 30 日修訂此公告？**  
Microsoft 修訂此公告的目的，是為了宣佈一項偵測邏輯變更，以修正 2010 年 6 月 8 日的 Microsoft .NET Framework 3.5 資訊安全性更新 (KB982865) 問題，該更新在已安裝 Microsoft .NET Framework 3.5 Service Pack 1 的系統上被 Microsoft Baseline Security Analyzer (MBSA) 誤列為「已安裝」。 儘管如此，MBSA 可正確偵測 Microsoft .NET Framework 2.0 Service Pack 2 和 Microsoft .NET Framework 3.5 Service Pack 1 資訊安全性更新 (KB979909) 的適用性。 此變更僅屬偵測變更，並不會對初始更新中的檔案造成影響。 已成功更新系統的客戶不需要重新安裝此更新。

此偵測變更可能會導致 KB982865 更新對使用 Windows Server Update Services (WSUS) 的客戶顯示為新的更新。 如果您尚未設置自動核准規則，您可能必須手動重新核准此更新。 未重新核准此更新，可能會導致此更新持續被提供給系統 (即使此更新並不適用)。 如需有關此偵測變更的詳細資訊，請參閱 [Microsoft 知識庫文件編號 894199](http://support.microsoft.com/kb/894199)。

**什麼是資料竄改弱點？**  
在資訊安全性中，資料竄改弱點可能會允許資料的惡意修改。 範例包括對持續資料 (如：資料庫中的資料) 所進行的未授權變更，以及資料透過開放式網路 (如：網際網路) 在兩部電腦之間流動時的變更。 特別針對這個弱點，如果使用有效的密碼編譯簽章對資料進行數位簽署，且此資料在簽署後遭到修改，然後數位簽章的驗證失敗，即表示發生竄改。 如果儘管事實上資料已遭竄改，而簽章的驗證仍成功，這也表示發生竄改。

**我該如何分辨已安裝的 Microsoft .NET Framework 版本？**  
您可以在一個系統上安裝和執行多個版本的 .NET Framework，且能以任何順序安裝各版本。 有數種方法能分辨目前已安裝的 .NET Framework 版本。 如需更多資訊，請參閱 [Microsoft 知識庫文件編號 318785](http://support.microsoft.com/kb/318785/zh-tw) 或 MSDN 文章，[分辨已安裝的 .NET Framework 版本](http://msdn.microsoft.com/en-us/library/y549e41e.aspx) (英文)。

**為什麼某些受支援作業系統上的 Microsoft .NET Framework 3.5 及 Microsoft .NET Framework 3.5 Service Pack 1 會受影響？**  
Microsoft .NET Framework 3.5 內含 Microsoft .NET Framework 2.0 Service Pack 1 與 Microsoft .NET Framework 3.0 Service Pack 1 的子元件。 Microsoft .NET Framework 3.5 Service Pack 1 內含 Microsoft .NET Framework 2.0 Service Pack 2 與 Microsoft .NET Framework 3.0 Service Pack 2 的子元件。 在部分作業系統上，Microsoft .NET Framework 3.5 及 Microsoft .NET Framework 3.5 Service Pack 1 可能安裝了含有弱點的子元件。

**我系統上安裝的 Microsoft .NET Framework 版本未列於此公告中。 我的設定是否會受此弱點影響？**  
本公告所列出的受影響軟體版本已經過測試判斷哪些版本會受到影響。 其他版本已超過它們的支援週期或不受支援。 在系統上安裝不受支援的 Microsoft .NET Framework 版本之客戶，建議將該版本的 Microsoft .NET Framework 解除安裝，並升級至較新版的 Microsoft .NET Framework。

Microsoft .NET Framework 可透過 \[控制台\] 中的 \[新增或移除程式\] 工具解除安裝。 如需更多關於移除特定 Microsoft .NET Framework 版本的資訊，請參閱 [Microsoft 知識庫文件編號 320122](http://support.microsoft.com/kb/320112/zh-tw)、[Microsoft 知識庫文件編號 824643](http://support.microsoft.com/kb/824643/zh-tw) 和 [Microsoft 知識庫文件編號 908077](http://support.microsoft.com/kb/908077/zh-tw)。

**此更新程式是否包含功能上的任何安全性相關變更？**  
是。 除了本公告＜弱點資訊＞一節中列出的變更外，此更新還包含對 ASP.NET 要求驗證功能的深度防禦變更，以解決一項弱點 - 該項弱點會允許攻擊者略過 ASP.NET 網站預設啟用的基本深度防禦措施。

引起此問題的原因在於 ASP.NET 中的要求驗證功能未正確檢查特定的字元序列。 ASP.NET 要求驗證功能無法取代限制不受信任輸入變數的有效驗證層。 開發人員若要進一步瞭解 ASP.NET 為 Web 應用程式提供的安全性功能，請參閱 MSDN 文章：[善用 ASP.NET 內建功能來擊退網路攻擊](http://msdn2.microsoft.com/en-us/library/ms972969.aspx)。

如果網站沒有有效的驗證層來限制不信任的使用者輸入，成功利用此弱點的攻擊者也許便能將任意內容 (包括 ASP.NET 內容) 注入受影響的網站。 這個深度防禦變更可緩和未公開報告的弱點。

**什麼是 ASP.NET 要求驗證？**  
對於 ASP.NET 要求驗證功能的深度防禦變更會在執行 ASP.NET 的網站上執行基本的輸入驗證。但是，ASP.NET 要求驗證功能無法取代可限制不信任之輸入變數的有效驗證層。 開發人員若要進一步瞭解 ASP.NET 為 Web 應用程式提供的安全性功能，請參閱 MSDN 文章：[善用 ASP.NET 內建功能來擊退網路攻擊](http://msdn2.microsoft.com/en-us/library/ms972969.aspx)，以及

**如何知道我的 ASP.NET Web 應用程式是否會受到此深度防禦變更的影響？**  
將所有不信任的輸入變數限制在預期值或字元範圍之內的 ASP.NET 型 Web 應用程式不會受到影響。 如需更多有關強化 ASP.NET Web 應用程式的資訊，請參閱 [Microsoft 知識庫文件編號 815155](http://support.microsoft.com/kb/815155/zh-tw)。

**什麼是 ASP.NET？**  
ASP.NET 是 Microsoft .NET Framework 中的技術集合，讓開發者能建立 Web 應用程式與 XML Web 服務。

傳統式網頁使用靜態 HTML 與指令碼的組合，但 ASP.NET 使用的是已編譯、事件驅動網頁。 因為 ASP.NET 為一網頁式應用程式環境，並需要基礎網頁伺服器提供基本的 HTTP 功能，因此 ASP.NET 是在 Internet Information Services (IIS) 之上執行。 如需更多資訊，請參閱[官方 Microsoft ASP.NET 網站](http://www.asp.net/) (英文)。

**為什麼將此問題視為深度防禦措施來處理？**  
ASP.NET 要求驗證被視為深度防禦變更來處理，可作為開發人員本身的輸入驗證之外的額外預防措施。 只有開發人員能定義什麼可構成特定應用程式的適當輸入。 即使多層這種防禦能有效協助防止攻擊者危及系統安全性，仍不宜依賴這些深度防禦功能。 因此，如果 ASP.NET 要求驗證遺漏了特定的字元序列，此問題的最大安全性影響並不會高於如果這個功能完全停用的狀況。

**什麼是深度防禦？**  
就資訊安全而言，深度防禦指的是一種建構多層防線，以協助防堵攻擊者危害網路或系統安全性的一種方法。

XML 簽章 HMAC 截斷驗證略過弱點 - CVE-2009-0217
----------------------------------------------

Microsoft .NET Framework 存在資料竄改弱點，可能會允許攻擊者竄改已簽署的 XML 內容，而且不會被偵測到。 這對自訂應用程式的安全性影響多寡，取決於特定的使用狀況。 如果已簽署的 XML 訊息是透過安全通道 (例如 SSL) 傳輸，便不會受到此弱點影響。

若要在「一般性弱點」清單標準條目中檢視此弱點，請參閱 [CVE-2009-0217](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0217) (英文)。

#### XML 簽章 HMAC 截斷驗證略過弱點 - CVE-2009-0217 的緩和因素

緩和因素是指存在於預設狀態中的設定、共用設定或一般最佳作法，可能會減少弱點影響的嚴重性。 下列緩和因素可能對您的狀況有所助益：

-   Microsoft Office 實作 XML 簽章檢查的方式，可讓攻擊者無法嘗試利用此資料竄改弱點。

#### XML 簽章 HMAC 截斷驗證略過弱點 - CVE-2009-0217 的因應措施

因應措施指的是無法徹底修正弱點，但有助於在套用更新之前封鎖已知攻擊模式的設定變更。 Microsoft 測試了下列因應措施和狀態，討論因應措施是否會降低功能：

-   **使用安全通道傳輸機密資訊**

    如果您的應用程式傳輸的是機密的 XML 資料或對應用程式全性非常重要的 XML 資料，請使用安全通道傳輸此資訊。 例如，如果您使用 Internet Information Server，則需要為機密或安全性重大的應用程式使用 SSL。

#### XML 簽章 HMAC 截斷驗證略過弱點 - CVE-2009-0217 的常見問題集

**這個弱點的範圍為何？**  
此弱點是可用於略過密碼編譯簽章的資料竄改弱點。

**造成這個弱點的原因為何？**  
引起此弱點的原因在於，W3C XML 簽章語法和處理 (XMLDsig) 建議在 Microsoft .NET Framework 中的實作方式。 如需更多關於 XMLDsig 的資訊，請參閱 W3C 建議，[XML 簽章語法和處理 (第二版)](http://www.w3.org/tr/xmldsig-core/) (英文) 或參閱 [XML 簽章的 Errata 第二版](http://www.w3.org/2008/06/xmldsigcore-errata.html) (英文) 內的「E03 HMAC 截斷 (CVE-2009-0217)」一節。

**什麼是 Microsoft .NET Framework？**  
Microsoft .NET Framework 是 Microsoft Windows 作業系統的一項元件，可用於建立及執行軟體應用程式與 Web 服務。 內容包括用於 Web 服務和 Web 應用程式 (ASP.NET) 的技術、資料存取 (ADO.NET)、智慧型用戶端應用程式 (Windows Forms)，以及許多其他技術。

**什麼是 HMAC？**  
如果傳送者與接收者共用秘密金鑰，則藉由驗證訊息的完整性與真實性，雜湊型訊息驗證碼 (HMAC) 可用於判定透過不安全通道傳送的訊息是否已遭竄改。

**什麼是 XML？**  
可延伸標記語言 (Extensible Markup Language，XML) 是一種標記語言，提供描述結構化資料的格式。 可延伸標記語言 (XML) 是一項 World Wide Web Consortium (W3C) 規格，以及標準通用標記語言 (SGML) 的一項子集。 如需更多關於 XML 的資訊，請參閱 MSDN 資料開發人員中心網頁 [XML 首頁](http://msdn.microsoft.com/xml/)或 W3C 建議：[可延伸標記語言 (XML) 1.0 (第五版)](http://www.w3.org/tr/rec-xml/) (英文)。

**攻擊者可能會利用這項弱點採取什麼行動？**  
成功利用此弱點的攻擊者可以略過某些密碼編譯簽章，進而竄改已簽署的 XML 內容，而且接收者偵測不到該變更。 如果訊息是從其原始意義徹底變更，這也可能構成偽造。 如果開發人員撰寫的應用程式依賴 HMAC 已簽署的 XML 內容，則對於安全性影響多寡，取決於該應用程式的特定使用狀況。

**攻擊者如何利用這項弱點？**  
攻擊者必須將蓄意製作的 XML 內容傳送至易受影響的系統。 Microsoft Office 實作 XML 簽章檢查的方式，不會遭攻擊者利用。

**因為這個弱點而承受風險的主要系統有哪些？**  
依賴雜湊型訊息驗證碼 (HMAC) 進行密碼編譯作業的系統或應用程式，其風險最高。 Microsoft Office 實作 XML 簽章檢查的方式，可讓攻擊者無法嘗試利用該弱點。

**更新的作用何在？**  
此更新利用 XMLDsig 建議在 Microsoft .NET Framework 中的實作方式。

**當資訊安全公告發行時，這項弱點是否已被揭發出來？**  
是。 這項弱點已經遭到公開揭發。 這項弱點已被指派「一般性弱點」編號 [CVE-2009-0217](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0217)。

**當本資訊安全公告發行時，Microsoft 是否已接獲任何消息，指出這項弱點已遭有心人士利用？**  
否。 當本資訊安全公告初次發行時，Microsoft 尚未接到任何有關本弱點已公開用來攻擊客戶的消息。

Application Compatibility Toolkit (ACT) 包含必要的工具和文件，可讓您在環境中部署 Microsoft Windows Vista、Windows Update、Microsoft 資訊安全性更新或新版 Windows Internet Explorer 之前，評估及減輕應用程式相容性問題。

**確認已經套用更新**

-   **Microsoft Baseline Security Analyzer**

    您可以使用 Microsoft Baseline Security Analyzer (MBSA) 工具確認受影響系統上是否已經套用資訊安全性更新程式。 如需詳細資訊，請參閱本公告前段的＜偵測與部署工具及指南＞一節。

-   **檔案版本驗證**

    由於 Microsoft Windows 的版本眾多，您電腦上實際執行的步驟可能會與此處描述的不同。 如遇到不同的狀況，請參閱產品的說明文件以完成這些步驟。

    1.  按一下 \[開始\]，然後按一下 \[搜尋\]。
    2.  在 \[搜尋結果\] 窗格中，在 \[搜尋小幫手\] 下按一下 \[所有檔案和資料夾\]。
    3.  在 \[部份或完整的檔案名稱\] 方塊中，輸入適當檔案資訊表中的檔案名稱，再按一下 \[搜尋\]。
    4.  在檔案清單中，用滑鼠右鍵按一下適當檔案資訊表中的某個檔案名稱，再按 \[內容\]。
        **注意：**視系統所安裝的作業系統或程式的版本而定，檔案資訊表中列出的檔案未必會全部安裝。
    5.  在 \[版本\] 索引標籤上，比較檔案版本與適當檔案資訊表中記錄的版本，判斷您系統上安裝的檔案版本。
        **注意：**在安裝時，檔案版本以外的屬性可能會變更。 若要驗證是否成功套用更新程式，利用檔案資訊表比對其他檔案屬性並不是妥當的做法。 此外，在某些情況下，檔案的名稱在安裝時可能會有所變更。 如果缺少檔案或版本資訊，請採用其他可用的方法來驗證更新程式的安裝情形。

-   **登錄機碼驗證**

    您也可以查看本節「參考表」中所列的登錄機碼，確認此資訊安全性更新所安裝的檔案。

    這些登錄機碼可能未包含完整的安裝檔案清單。 而且，當系統管理員或 OEM 將資訊安全性更新整合或匯集到 Windows 安裝原始檔時，可能無法正確建立這些登錄機碼。

</td>

<td style="border:1px solid black;">

<td style="border:1px solid black;">

</td>

不先提示，直接覆寫 OEM 檔案。
</td>
</tr>
<tr>
<td style="border:1px solid black;">
**/nobackup**
</td>
<td style="border:1px solid black;">
不備份解除安裝所需的檔案。

<td style="border:1px solid black;">

</td>

<td style="border:1px solid black;">

</td>
</tr>
<tr class="alternateRow">
<td style="border:1px solid black;">
**/integrate:path**
</td>
<td style="border:1px solid black;">

</td>
</tr>
<tr>
<td style="border:1px solid black;">
**/extract\[:path\]**
</td>
<td style="border:1px solid black;">
不啟動安裝程式，僅解壓縮檔案。
</td>
</tr>
<tr class="alternateRow">
<td style="border:1px solid black;">
**/ER**
</td>
<td style="border:1px solid black;">
啟用延伸的錯誤報告。
</td>
</tr>
<tr>
<td style="border:1px solid black;">
**/verbose**
</td>
<td style="border:1px solid black;">
啟用詳細資訊記錄。 在安裝時建立 %Windir%\\CabBuild.log。此記錄檔詳細說明複製的檔案。 使用這個參數可能會使安裝速度慢很多。
</td>
</tr>
</table>

**確認已經套用更新**

-   **Microsoft Baseline Security Analyzer**

    您可以使用 Microsoft Baseline Security Analyzer (MBSA) 工具確認受影響系統上是否已經套用資訊安全性更新程式。 如需詳細資訊，請參閱本公告前段的＜偵測與部署工具及指南＞一節。

-   **檔案版本驗證**

    由於 Microsoft Windows 的版本眾多，您電腦上實際執行的步驟可能會與此處描述的不同。 如遇到不同的狀況，請參閱產品的說明文件以完成這些步驟。

    1.  按一下 \[開始\]，然後按一下 \[搜尋\]。
    2.  在 \[搜尋結果\] 窗格中，在 \[搜尋小幫手\] 下按一下 \[所有檔案和資料夾\]。
    3.  在 \[部份或完整的檔案名稱\] 方塊中，輸入適當檔案資訊表中的檔案名稱，再按一下 \[搜尋\]。
    4.  在檔案清單中，用滑鼠右鍵按一下適當檔案資訊表中的某個檔案名稱，再按 \[內容\]。
        **注意：**視系統所安裝的作業系統或程式的版本而定，檔案資訊表中列出的檔案未必會全部安裝。
    5.  在 \[版本\] 索引標籤上，比較檔案版本與適當檔案資訊表中記錄的版本，判斷您系統上安裝的檔案版本。
        **注意：**在安裝時，檔案版本以外的屬性可能會變更。 若要驗證是否成功套用更新程式，利用檔案資訊表比對其他檔案屬性並不是妥當的做法。 此外，在某些情況下，檔案的名稱在安裝時可能會有所變更。 如果缺少檔案或版本資訊，請採用其他可用的方法來驗證更新程式的安裝情形。

-   **登錄機碼驗證**

    您也可以查看本節「參考表」中所列的登錄機碼，確認此資訊安全性更新所安裝的檔案。

    這些登錄機碼可能未包含完整的安裝檔案清單。 而且，當系統管理員或 OEM 將資訊安全性更新整合或匯集到 Windows 安裝原始檔時，可能無法正確建立這些登錄機碼。

**確認已經套用更新**

-   **Microsoft Baseline Security Analyzer**

    您可以使用 Microsoft Baseline Security Analyzer (MBSA) 工具確認受影響系統上是否已經套用資訊安全性更新程式。 如需詳細資訊，請參閱本公告前段的＜偵測與部署工具及指南＞一節。

-   **檔案版本驗證**

    由於 Microsoft Windows 的版本眾多，您電腦上實際執行的步驟可能會與此處描述的不同。 如遇到不同的狀況，請參閱產品的說明文件以完成這些步驟。

    1.  按一下 \[開始\]，然後按一下 \[搜尋\]。
    2.  在 \[搜尋結果\] 窗格中，在 \[搜尋小幫手\] 下按一下 \[所有檔案和資料夾\]。
    3.  在 \[部份或完整的檔案名稱\] 方塊中，輸入適當檔案資訊表中的檔案名稱，再按一下 \[搜尋\]。
    4.  在檔案清單中，用滑鼠右鍵按一下適當檔案資訊表中的某個檔案名稱，再按 \[內容\]。
        **注意：**視系統所安裝的作業系統或程式的版本而定，檔案資訊表中列出的檔案未必會全部安裝。
    5.  在 \[版本\] 索引標籤上，比較檔案版本與適當檔案資訊表中記錄的版本，判斷您系統上安裝的檔案版本。
        **注意：**在安裝時，檔案版本以外的屬性可能會變更。 若要驗證是否成功套用更新程式，利用檔案資訊表比對其他檔案屬性並不是妥當的做法。 此外，在某些情況下，檔案的名稱在安裝時可能會有所變更。 如果缺少檔案或版本資訊，請採用其他可用的方法來驗證更新程式的安裝情形。

-   **登錄機碼驗證**

    您也可以查看本節「參考表」中所列的登錄機碼，確認此資訊安全性更新所安裝的檔案。

    這些登錄機碼可能未包含完整的安裝檔案清單。 而且，當系統管理員或 OEM 將資訊安全性更新整合或匯集到 Windows 安裝原始檔時，可能無法正確建立這些登錄機碼。

    由於 Microsoft Windows 的版本眾多，您電腦上實際執行的步驟可能會與此處描述的不同。 如遇到不同的狀況，請參閱產品的說明文件以完成這些步驟。

    4.  您也可以按一下 \[詳細資料\] 索引標籤，然後將檔案版本與修改日期等資訊與公告 KB 文件提供的檔案資訊表進行比對。
    5.  最後，您也可以按一下 \[舊版\]，將舊版檔案的檔案資訊，與檔案的新或更新版本之檔案資訊進行比較。
