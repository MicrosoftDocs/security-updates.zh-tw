Security Bulletin

Microsoft Security Bulletin MS10-085 - 重要
===========================================

SChannel 中的資訊安全風險可能會允許拒絕服務 (2207566)
-----------------------------------------------------

發行: 2010年10月13日

**版本:** 1.0

### 一般資訊

#### 提要

這個資訊安全更新可解決 Windows 安全通道 (Schannel) 安全性套件中一項未公開報告的資訊安全風險。 如果主控啟用安全通訊端層 (SSL) 功能網站的受影響 Internet Information Services (IIS) 伺服器收到蓄意製作的封包訊息，此資訊安全風險可能會允許拒絕服務。 預設並不設定 IIS 來主控 SSL 網站。

對於所有受支援版本的 Windows Vista、Windows Server 2008、Windows 7 與 Windows Server 2008 R2，此資訊安全更新的等級為「重要」。如需詳細資訊，請參閱本節中的＜受影響及不受影響的軟體＞小節。

此資訊安全更新藉由變更該 IIS 處理用戶端憑證要求的方式，來解決此資訊安全風險。 如需更多有關資訊安全風險的資訊，請參閱下節＜資訊安全風險資訊＞下的特定資訊安全風險項目的＜常見問題集 (FAQ)＞小節。

**建議。** 大部分客戶都已啟用自動更新，並且不必須採取任何行動，因為資訊安全更新將自動下載和安裝。 沒有啟用自動更新的客戶則必須檢查更新，並手動安裝更新。 如需有關自動更新中特定設定選項的資訊，請參閱 [Microsoft 知識庫文件編號 294871](http://support.microsoft.com/kb/294871/zh-tw)。

若是系統管理員和企業安裝，或是想要手動安裝此資訊安全更新的使用者，Microsoft 建議客戶使用更新管理軟體，立即套用更新，或使用 [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747) 服務檢查更新。

另請參閱本公告下文的＜偵測與部署工具及指南＞章節。

**已知問題。** 無

#### 受影響及不受影響的軟體

下列軟體已經過測試判斷哪些版號或版本會受到影響。 其他版本超過它們的支援週期或不受影響。 若要瞭解您的軟體版本的支援週期，請造訪 [Microsoft 產品技術支援週期網站](http://go.microsoft.com/fwlink/?linkid=21742)。

TLSv1 拒絕服務資訊安全風險 - CVE-2010-3229  
------------------------------------------
  
<span></span>
在 Windows Server 2008 和 Windows Vista 的 Internet Information Services (IIS) 7.0，以及在 Windows Server 2008 R2 和 Windows 7 的 IIS 7.5 上實作 SChannel 程序用戶端憑證的方式中，存在拒絕服務資訊安全風險。遠端的匿名攻擊者可將蓄意製作的網路封包傳送至受影響系統，而導致 LSASS 服務停止回應並重新啟動系統。 唯有在啟用 SSL 時 (這並非預設的設定)，系統才會受影響。
  
若要在「一般性資訊安全風險」清單標準條目中檢視此資訊安全風險，請參閱 [CVE-2010-3229](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2010-3229) (英文)。
  
#### TLSv1 拒絕服務資訊安全風險 - CVE-2010-3229 的緩和因素
  
緩和因素是指存在於預設狀態中的設定、共用設定或一般最佳作法，可能會減少資訊安全風險影響的嚴重性。 下列緩和因素可能對您的狀況有所助益：
  
-   預設並不設定讓 IIS 接受 SSL 連線。  
-   攻擊成功後會自動重新啟動系統，讓系統復原。
  
#### TLSv1 拒絕服務資訊安全風險 - CVE-2010-3229 的因應措施
  
因應措施指的是無法徹底修正資訊安全風險，但有助於在套用更新之前封鎖已知攻擊模式的設定變更。 Microsoft 測試了下列因應措施和狀態，討論因應措施是否會降低功能：
  
-   **在 IIS 中停止 SSL 網站**
  
    執行下列步驟，停止在 IIS 伺服器上所主控啟用 SSL 的網站：
  
    1.  開啟 \[Internet Information Services (IIS) Manager\]  
    2.  在左側的 \[連線\] 面板中，選取使用 SLL 的網站  
    3.  。按一下右面板中的 \[停止\]  
    4.  。對每個啟用 SSL 的網站重複步驟 2 和 3
  
    **因應措施的影響。** 用戶端只能存取未啟用 SSL 網站
  
    **如何復原因應措施：**
  
    1.  開啟 \[Internet Information Services (IIS) Manager\]  
    2.  在左側的 \[連線\] 面板中，選取使用 SLL 的網站  
    3.  按一下右面板中的 \[啟動\]

#### TLSv1 拒絕服務資訊安全風險 - CVE-2010-3229 的常見問題集
  
**這個資訊安全風險的範圍為何？**    
這是一項拒絕服務的資訊安全風險。 利用此資訊安全風險的攻擊者可能致使 LSASS 服務停止及重新啟動電腦，而在系統重新啟動完畢之前會持續出現拒絕服務的情況。 然後攻擊者便可再次利用此資訊安全風險。
  
**造成這個資訊安全風險的原因為何？**    
當 SChannel 無法在 SSL/TLS 交握期間先檢查邏輯條件，再處理通訊協定要求時，就會造成此資訊安全風險。
  
**什麼是安全通道 (SChannel) 安全性套件？**    
安全通道 (Schannel) 安全性套件是執行安全通訊端層 (SSL) 和傳輸層安全性 (TLS) 網際網路標準驗證通訊協定的資訊安全支援提供者 (SSP)。 在實作安全通訊之中就會用這些元件來支援網頁瀏覽等常見的網際網路及網路應用程式。 Schannel 是協助提供驗證服務，以在用戶端和伺服器之間提供安全通訊之安全性套件的一部分。
  
如需詳細資訊，請參閱 MSDN 文章：[Secure Channel](http://msdn.microsoft.com/en-us/library/aa380123(vs.85).aspx) (英文)。
  
**什麼是 TLS 和 SSL？**    
傳輸層安全性 (TLS) 交握通訊協定負責建立或恢復安全工作階段的必備驗證及金鑰交換。 安全通訊端層 (SSL) 是「傳輸層安全性」通訊協定的前身。 透過公開金鑰與秘密金鑰技術的結合，TLS 和 SSL 可以執行相同的功能並支援安全的網路通訊。
  
如需更多資訊，請參閱 TechNet 的文章：[TLS/SSL 如何運作](http://technet.microsoft.com/en-us/library/cc783349(ws.10).aspx) (英文)。
  
**什麼是 TLS 或 SSL 交握？**    
使用 TLS 或 SSL 建立安全工作階段時，交握通訊協定會管理下列項目：
  
-   加密套件交涉  
-   伺服器及用戶端 (選擇性) 的驗證  
-   工作階段金鑰資訊交換
  
如需更多資訊，請參閱 TechNet 的文章：[TLS/SSL 如何運作](http://technet.microsoft.com/en-us/library/cc783349(ws.10).aspx) (英文)。
  
**我能否將 IIS 中的 \[接受用戶端憑證\] 設定作為因應措施使用？**    
否。 如果系統正在啟用 SSL 的情況下執行受影響的軟體，則在 IIS 中將 \[接受用戶端憑證 \] 設定設為 \[忽略\] 也無法有效防止利用此資訊安全風險。
  
**攻擊者可能會利用這項資訊安全風險採取什麼行動？**    
成功利用此資訊安全風險的攻擊者將導致伺服器停止回應，並且需要重新啟動。
  
**攻擊者如何利用這項資訊安全風險？**    
遠端的匿名攻擊者可將蓄意製作的封包訊息傳送至主控啟用 SSL 的網站之受影響 IIS 伺服器，如此可能會導致 LSASS 停止回應，進而拒絕服務。
  
**因為這個資訊安全風險而承受風險的主要系統有哪些？**    
此資訊安全風險會影響啟用 SSL 的網頁伺服器。
  
**更新的作用何在？**    
此更新藉由變更 IIS 處理用戶端憑證要求的方式，來解決此資訊安全風險。
  
**當資訊安全公告發行時，這項資訊安全風險是否已揭發出來？**    
否。 Microsoft 是經由協同合作的來源接獲有關這項資訊安全風險的訊息。
  
**當本資訊安全公告發行時，Microsoft 是否已接獲任何消息，指出這項資訊安全風險已遭有心人士利用？**    
否。 當本資訊安全公告初次發行時，Microsoft 尚未接到任何有關本資訊安全風險已公開用來攻擊客戶的消息。
